# Board Template System Architecture

## Overview
The FlowSpace Template System allows users to create, share, and reuse predefined board structures (Flowcharts, Mindmaps, Wireframe kits). This system reduces friction for new users and standardizes workflows across teams.

Templates reuse the authoritative **.flowspace** schema to ensure full fidelity of nodes, edges, and metadata.

---

## 1. Entities

### BoardTemplate
Represents a reusable blueprint for a board.

| Property | Type | Description |
| :--- | :--- | :--- |
| **Id** | UUID | Unique identifier (UUIDv7). |
| **Name** | String | Template name (e.g., "Software Architecture", "Kanban Board"). |
| **Description** | String | Detailed explanation of the template's purpose. |
| **ThumbnailUrl** | String | URL to a generated PNG preview in MinIO. |
| **BoardType** | Enum | `Whiteboard`, `Flowchart`, `Mindmap`, `Wireframe`. |
| **IsSystem** | Boolean | True if provided by FlowSpace; false if user-created. |
| **CreatedBy** | UUID | Reference to the User who created the template (nullable for system). |
| **Content** | JSONB | The actual .flowspace schema (Nodes, Edges, Viewport). |
| **CreatedAt** | DateTime | Timestamp. |

## 2. Core Features

### Create Template
Users can promote an existing board to a template.
1. **Source**: User selects a board they own.
2. **Snapshot**: The system generates a `.flowspace` JSON object from the current active nodes and edges.
3. **Storage**: A new `BoardTemplate` record is created.
4. **Visibility**: User-created templates are initially private to the user's account.

### Use Template (Instantiation)
Creates a fresh board within a workspace based on the template.
1. **Target**: User selects a Workspace and a Template.
2. **Deep Copy**:
    - A new `Board` is created in the database.
    - All `nodes` from the template's `Content.nodes` are batch-inserted with new UUIDs.
    - All `edges` from the template's `Content.edges` are batch-inserted, with source/target IDs re-mapped to the new node UUIDs.
3. **Initialization**: The UI opens the new board and initializes the Yjs state vector from the template data.

### Duplicate Template
Allows users to fork a System or Shared template to create their own variation.
1. Create a copy of the `BoardTemplate` record.
2. Assign the current `UserId` to `CreatedBy`.
3. Clear the `IsSystem` flag.

## 3. API Specification

### GET `/api/v1/templates`
List available templates.
- **Query**: `?type=...&source=[system|user|all]`
- **Response**: `List<BoardTemplateResponse>`

### POST `/api/v1/templates`
Create a template from an existing board.
- **Request**: `{ "boardId": "uuid", "name": "...", "description": "..." }`
- **Response**: `BoardTemplateResponse`

### POST `/api/v1/workspaces/{workspaceId}/boards/from-template`
Instantiate a board from a template.
- **Request**: `{ "templateId": "uuid", "name": "..." }`
- **Response**: `BoardResponse`

---

## 4. Integration with UI

### Template Gallery
A specialized view in the Dashboard where users can browse categorized templates.
- **Categories**: "Popular", "Flowcharts", "System Design", "Agile".
- **Previews**: Hovering over a template card displays the `ThumbnailUrl` (a high-quality PNG generated by the **Export Worker**).

### Instantiation Flow
1. User clicks "New Board" in a Workspace.
2. The UI offers "Blank Board" or "Start from Template".
3. Choosing a template triggers the `from-template` API, redirecting the user to the newly created Canvas.

---

## 5. Scalability Considerations

- **Storage**: Since templates are stored as `JSONB`, very large templates (> 5,000 nodes) should be compressed (Gzip) and stored in `bytea` if storage cost becomes a concern.
- **Concurrency**: Template instantiation uses the same batch-insertion logic as **Restoring a Version**, ensuring it's highly performant even for complex kits.
