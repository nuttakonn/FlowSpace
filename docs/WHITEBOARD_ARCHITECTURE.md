# Whiteboard Architecture

## Overview
The FlowSpace Whiteboard is a free-form creative visual engine built on **tldraw**. It supports an infinite canvas, freehand drawing, shapes, and complex object relationships (connectors) while maintaining the platform's standard real-time collaboration and versioning guarantees.

---

## 1. Core Technology Stack
- **tldraw (v2+)**: The primary rendering and interaction library for the whiteboard.
- **Yjs**: CRDT engine for managing the shared whiteboard state.
- **y-tldraw**: Bridge library for synchronizing tldraw records with Yjs shared types.
- **SignalR**: Transport layer for binary Yjs updates (via the existing `CollaborationHub`).

---

## 2. Document Model & Mapping

To maintain interoperability with the FlowSpace backend and culling systems, tldraw records are mapped to the platform's core entities.

### Entity Mapping
| tldraw Record | FlowSpace Entity | Description |
| :--- | :--- | :--- |
| `TLShape` (Geo) | `Node` | Rectangles, Circles, Arrows, etc. |
| `TLShape` (Draw) | `Node` | Freehand lines. Path data is stored in the `Metadata` JSONB column. |
| `TLShape` (Note) | `Node` | Sticky notes. |
| `TLRelationship` | `Edge` | Logical connectors between shapes. |
| `TLAsset` | `Blob/S3` | Images and external assets linked via metadata. |

### Metadata Schema
Specialized tldraw properties (opacity, color, font, dash style) are stored in the `Node.Metadata` JSONB column to ensure no data loss during round-trips.

---

## 3. Real-time Synchronization Flow

1. **Store-to-Yjs Bridge**:
    - The frontend uses the `useSync` hook from `tldraw`.
    - Every local mutation in the tldraw store is converted to a Yjs update.
2. **Binary Transport**:
    - The Yjs update is sent via SignalR `UpdateCanvas` to the backend.
    - The backend relays the update to other participants in the board group.
3. **Presence (Awareness)**:
    - User cursors and active selections use the **Yjs Awareness** protocol.
    - tldraw's `Editor.user` state is synced across the network to show remote collaborators' actions.

---

## 4. Integration with Platform Systems

### Versioning & Snapshots
- **Snapshots**: The `CanvasSnapshot` entity captures the entire tldraw state as a `ByteA` binary blob (the Yjs state vector).
- **History**: Local undo/redo stack in tldraw is bounded to 100 steps. Global history is handled by `BoardVersion`.

### Infinite Canvas Persistence
- **Lazy Loading**: For extremely large whiteboards, the frontend implements a spatial index to fetch only the tldraw records (Nodes/Edges) within the current viewport plus a 20% buffer.
- **Auto-Save**: The backend "Headless Client" periodically flushes the Yjs state to the `nodes` and `edges` tables.

### Export System
- **Server-side**: The `CanvasExportWorker` (Playwright) targets a route that renders the tldraw canvas in `readonly` mode. It captures PNG/PDF/SVG.
- **Client-side**: Leverages tldraw's `exportToSvg` utility for instant, high-quality vector downloads of selected areas.

---

## 5. Key Features

- **Infinite Canvas**: Top-left origin (0,0) is used for coordinate normalization across formats.
- **Shapes & Connectors**: Supports arrows with smart routing that anchor to shapes.
- **Rich Text**: Collaborative text editing within sticky notes and text shapes using `Y.Text`.
- **Image Support**: Users can drag-and-drop images. Images are uploaded to MinIO and referenced in the Node metadata.

---

## 6. Security & RBAC
- **Permission Enforcement**: The `CollaborationHub` validates that the user has `Editor` permissions before accepting Yjs updates for the whiteboard.
- **Asset Protection**: Image assets linked in the metadata are only accessible via signed URLs generated by the API for authorized board members.
